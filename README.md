## О репозитории

**Tunnel LTE Universal Checked** — это публичный репозиторий с автоматически обновляемым списком VLESS-конфигураций, ориентированных на работу в условиях LTE whitelist-фильтрации.

Репозиторий создан специально для проекта **Tunnel**:

```text
https://github.com/hussaroff/tunnel
```

Основная цель репозитория — дать приложению Tunnel готовый пул предварительно проверенных VLESS-маршрутов, которые можно быстро загрузить, проверить уже на устройстве пользователя и использовать для подключения.

---

## Для чего это нужно

В сетях с whitelist-фильтрацией обычный подход часто не работает:

```text
нашёл VLESS → импортировал → нажал подключиться → не работает
```

Проблема в том, что доступность конфигурации зависит не только от самого сервера.

На работу влияют:

- мобильный оператор;
- регион;
- LTE / 3G / Wi-Fi;
- DNS;
- IP / CIDR;
- SNI;
- TLS / REALITY;
- WebSocket / TCP / XHTTP;
- DPI-фильтрация;
- маршрут от конкретного пользователя до сервера.

Поэтому Tunnel использует другой подход:

```text
загрузить пул
↓
проверить профили на устройстве пользователя
↓
найти реально ответивший маршрут
↓
подключиться
```

Этот репозиторий нужен именно для первого этапа — дать приложению большой, очищенный и предварительно проверенный пул.

## Почему рекомендуется Tunnel

Для использования этого списка рекомендуется приложение **Tunnel**:

```text
https://github.com/hussaroff/tunnel
```

Tunnel создаётся не как обычный технический клиент для ручного импорта конфигов, а как приложение, которое помогает найти рабочий маршрут в сложных сетевых условиях.

Tunnel может использовать:

- автоматическую загрузку конфигураций;
- проверку доступности серверов;
- real-delay проверку;
- failover-логику;
- работу с whitelist-oriented пулами;
- пользовательскую обратную связь о работоспособности серверов;
- выбор подходящего маршрута на стороне устройства.

Обычный клиент может показать, что сервер имеет TCP-ping, но это ещё не значит, что VLESS реально работает. Tunnel ориентирован именно на практический сценарий: найти тот профиль, который отвечает в текущей сети пользователя.

## Как формируется список

Список формируется автоматически серверным сканером.

Общий цикл:

```text
источники VLESS
↓
загрузка всех доступных ссылок
↓
игнорирование неответивших источников
↓
извлечение только vless:// строк
↓
удаление дублей
↓
загрузка whitelist IP/CIDR
↓
локальная фильтрация по белым спискам
↓
real-delay проверка через Xray
↓
запись успешных конфигураций в pre-checked.txt
↓
замена старого checked.txt
↓
публикация результата в GitHub
```

Если источник временно не отвечает, он не ломает весь процесс.  
Сканер просто пропускает его и продолжает проверку остальных источников.

---

## Что значит checked

В этом репозитории `checked` означает:

```text
конфигурация была собрана из whitelist-oriented источника,
прошла фильтрацию по IP/CIDR белых списков,
и успешно ответила при серверной real-delay проверке через Xray.
```

Это сильнее, чем обычный ping.

Обычный ping или TCP-connect показывает только:

```text
сервер или порт отвечает
```

Но не проверяет:

- валидность UUID;
- VLESS-handshake;
- TLS;
- REALITY;
- public key;
- shortId;
- SNI;
- path;
- host header;
- XHTTP / WS / TCP параметры;
- возможность реального выхода через туннель.

Поэтому в этом репозитории используется именно real-delay подход.

---

## Почему серверная проверка не равна гарантии работы

Важно: `checked.txt` — это не абсолютная гарантия, что каждый профиль будет работать у каждого пользователя.

Серверная проверка выполняется с одного сервера и одной сети.  
А пользователь подключается из другой среды:

```text
телефон
↓
LTE / Wi-Fi
↓
оператор
↓
региональная маршрутизация
↓
DPI / whitelist-фильтрация
↓
VLESS-сервер
```

Поэтому возможны ситуации:

```text
работает на Wi-Fi, но не работает на LTE
работает у одного оператора, но не работает у другого
работает на серверной проверке, но не отвечает у пользователя
имеет TCP-ping, но не проходит VLESS/REALITY
```

Именно поэтому финальная проверка должна происходить в Tunnel уже на устройстве пользователя.

## Файлы репозитория

### `checked.txt`

Основной файл подписки.

Содержит VLESS URI:

```text
vless://uuid@host:port?type=...&security=...#name
```

Строки, начинающиеся с `#`, являются комментариями.

---

### `stats.json`

Статистика последнего цикла проверки.

Может содержать:

- время генерации;
- количество источников;
- количество найденных VLESS;
- количество отфильтрованных конфигураций;
- количество успешно проверенных;
- параметры проверки;
- краткий список лучших ответивших маршрутов.

---

### `results.json`

Расширенный отчёт последней проверки.

Используется для анализа, отладки и дальнейшей автоматизации.

Может содержать:

- исходную ссылку;
- host;
- port;
- network;
- security;
- sni;
- задержку;
- ошибки;
- источник конфигурации;
- результат проверки.

## Логика Tunnel при whitelist-фильтрации

Tunnel должен работать не как простой импортёр конфигов, а как система выбора маршрута.

Правильная логика:

```text
скачать checked.txt
↓
загрузить пул в память
↓
проверить профили на текущей сети пользователя
↓
найти реально ответившие
↓
выбрать лучший по задержке / стабильности
↓
подключиться
```

Если профиль не отвечает на устройстве пользователя, он считается неподходящим для текущих условий.

Если профиль отвечает, он может использоваться как рабочий маршрут.

---

## Почему список обновляется полностью

Каждый цикл сканера создаёт новый `pre-checked.txt`.

После завершения проверки:

```text
старый checked.txt удаляется
pre-checked.txt переименовывается в checked.txt
обновлённый checked.txt публикуется в GitHub
```

Это нужно для того, чтобы в подписке не накапливались старые мёртвые конфигурации.

Репозиторий отражает актуальный результат последнего полного цикла проверки.
